# Spellshape Repository

## Purpose

Spellshape is an AI-native parametric design runtime that transforms natural-language design intent into structured, editable 3D workflows. Rather than generating geometry directly from prompts (which is unreliable), it uses a **structured DSL approach** where:

1. Prompts are parsed into structured facts
2. Facts are matched to reusable planning patterns called **"families"**
3. The system resolves legal plans and compiles them into executable action chains
4. Plans are validated and only then executed to generate 3D geometry in the browser

The heart of Spellshape is the **modeling briefs approach** - a framework that lets LLMs reason about 3D through a structured DSL of actions and families, then compile, validate, and execute those plans into geometry.

## Core Concepts

### Modeling Briefs Approach (Demo3 — primary)

**Demo3 (`/src/routes/demo3/+page.svelte`) is the primary pipeline.** It calls `planModelingBriefsLlmOnly` in `src/lib/modeling-briefs/api-llm-only.ts`, which POSTs to `/api/modeling-briefs/llm-only-plan` and runs the **LLM-only pipeline** at `src/lib/server/modeling-briefs/llm-only-pipeline.ts`. All new work on prompt design, operator cards, anomaly detection, and repair rules should target this pipeline.

Demo2 (`/src/routes/demo2/+page.svelte`) is kept as a working legacy surface for the older rule-based pipeline in `src/lib/server/modeling-briefs/pipeline.ts`. Changes to shared modules (compile-plan, browser executors, operator cards) must keep demo2 functional, but demo2 is no longer the target for new features.

The modeling briefs approach (as surfaced in demo3) works in these stages:

1. **Scene Brief**: Parse prompt into scene-level intent (objects, relations)
2. **Entity Decomposition**: LLM-driven decomposition of compound objects (e.g., "a clock") into functional sub-parts with explicit geometry, spatial relations, and consistent dimensions. Uses root-measure dimensional hierarchy: anchor entity's main dimension is the root, all other dimensions are expressed as ratios. `SceneIrLayoutSpec` carries `radius` and `spacing` for arrangement parameters.
3. **Object Strategy**: Decide build strategy (single-part vs multipart) and geometry mode
4. **Part Brief**: Define individual parts with constructor operators, placement, and repetition
5. **Compile**: Convert briefs to executable action chains
6. **Execute**: Run actions in browser to generate 3D geometry

#### Iterative Prompting

The modeling briefs approach supports **iterative prompts** - making modifications to an existing scene:
- First prompt: "a lamp on a table" → generates Table and Lamp
- Follow-up prompt: "make lamp red" → modifies the Lamp while preserving the Table

Key components:
- **LLM Reconciliation**: Uses LLM to determine which targets to preserve/modify in follow-up prompts
- **Target Matching**: Matches targets between prompts to preserve scene state
- **Force Regeneration**: Targets marked as "modified" by LLM are regenerated instead of reused

### Planning Pipeline

```
prompt → Lexical Parser → Facts → Candidate Retrieval → Strategy Selection 
       → Plan Family Selection → Resolved Plan Graph → Compiler 
       → Deterministic Rewrites → Validator → Execution Package → 3D Output
```

### Key Types

- **PlanningMode**: `objectCentric`, `layoutCentric`, `terrainCentric`, `deltaEdit`
- **PrimaryStrategy**: `assembly`, `singleHost`, `hybrid`
- **CanonicalStages**: `layoutGeneration` → `preStructure` → `structureGeneration` → `hostGeneration` → `postHost` → `postAssembly` → `finalization`

- **Dynamic operator retrieval for LLM prompts**: `src/lib/server/modeling-briefs/operator-retrieval.ts` filters which curated operators are surfaced to the LLM in Step 2 based on prompt keywords, object label, and Step 1 reason. A small always-include core (basic constructors, `arrange`, `mergeGroup`, `group`) is always in the prompt; deformers, simulations, and exotic constructors (`lathe`, `shell`, `generateText`, `landscape`, `sweep`, …) appear only when keyword cues match. Operator schemas and the curated allow-lists (`curated-operators.ts`) are unchanged — this is a prompt-layer filter only, so human-edited briefs can still invoke any curated op. `src/lib/server/modeling-briefs/test-operator-retrieval.ts` is a regression script you can run with `npx tsx src/lib/server/modeling-briefs/test-operator-retrieval.ts` to verify keyword coverage and core-set inclusion.
- **Rich `extrude` with profile discriminator and signed offset**: `extrude` now accepts a structured profile object (`{kind: 'rect' | 'roundedRect' | 'circle' | 'ngon' | 'freeform', ...per-kind params}`) extruded along +Y from y=0. A signed `offset` inflates the outer profile when `>0` or creates a uniform-thickness hollow wall of `|offset|` when `<0` (e.g. a phone body shell, a picture frame, a tube). Implementation lives in `src/lib/spellshape/geometry/profile-helpers.ts` (pure polygon math, CCW primitives + convex-polygon offset) and `createExtrudeFromProfileKind` in `src/lib/spellshape/action-registry-browser-executor.ts`. The legacy array-of-points form is still supported and keeps its original open-band behavior, so existing briefs compile unchanged. Operator card description in `src/lib/spellshape/cards/operator-cards.json` and `action-registry-data.json` is the source of truth for LLM-facing guidance. Regression: `npx tsx src/lib/spellshape/geometry/test-profile-helpers.ts`.
- **Data-driven rails via `bridgeAlong`**: `bridgeAlong` is the canonical constructor for any continuous rail/skirt/garland/wire that must ride through every instance of an arranged group (handrails on a spiral-staircase baluster run, garlands across hooks, ropes across posts). It reads per-instance world-space anchors stamped on the merged arrange output (`userData.__instances` — `{index, center, topCenter, bottomCenter, minY, maxY}` captured in `_mergeInstances` inside `src/lib/core/action-registry.js`), threads a `CatmullRomCurve3` (default) or polyline through them, and sweeps a profile (same discriminator as `extrude.profile`) along the curve with `THREE.ExtrudeGeometry`. The LLM authors the part with `constructor:"bridgeAlong"` and `params.source = "@<repeatedPartId>"`; the compiler post-pass `rewireBridgeAlongSources` in `src/lib/modeling-briefs/compile-plan.ts` retargets the ref from the pre-arrange seed alias to the final arrange output alias (e.g. `stair__baluster` → `stair__baluster__spiral`), so the LLM does not need to know the internal `__spiral`/`__radial` suffix. The rewire handles three advancement rules: (1) suffix-append for ordinary placements (`X → X__attached`), (2) cross-root `mergeGroup` consumption (`{X__attached, Y__attached, Z__attached} → Host__groupProto`), and (3) sibling-root advancement when an arrange consumes an intermediate derivative (`Host__groupProto → Host__spiral`) — this is what lets a bridgeAlong targeting a SATELLITE baluster inside a repeating stair group still reach the final 18-instance spiral output. Pure helpers (anchor sorting, polyline extension, helicity test) live in `src/lib/spellshape/geometry/bridge-along-helpers.ts`; the runtime op is `executeBridgeAlong` in `src/lib/spellshape/action-registry-browser-executor.ts`. Registration: `CURATED_CONSTRUCTORS` in `curated-operators.ts`, keywords (`handrail`, `railing`, `balustrade`, `garland`, `staircase`, `bridge`, `fence`, etc.) in `operator-retrieval.ts`, schema + description in `action-registry-data.json`, prompt rule 8 + repair rule 13 in `llm-only-pipeline.ts`. The LLM-only pipeline also detects any hand-authored helical `sweep` (pathPoints already helical, or straight sweep + wrap-style deformer) and rewrites it to `bridgeAlong` during repair. Regressions: `npx tsx src/lib/spellshape/geometry/test-bridge-along-helpers.ts` and `npx tsx src/lib/modeling-briefs/test-rewire-bridge-along.ts`.
- **Vertical structural cylinder guard**: Columns, shafts, posts, pillars, poles, masts, newels, stems, trunks, and spindles are conventionally upright. Four layers of defense keep them that way: (1) a proactive orientation-section note in the Step-2 prompt, (2) an anomaly detector in `collectLayoutAnomalies` (matches the short primitive name `"cylinder"` as well as the canonical `"generateCylinder"` since Step-2 output uses the short form) feeding repair rule 14, (3) a deterministic strip inside `convertStep2Parts` that removes `axis`/`direction` values of `"x"` or `"z"` on any cylinder whose id or label contains a structural keyword, and (4) a veto inside `compile-plan.ts`'s `detectConnectorAxis` / axle-axis force-apply loop that prevents the horizontal-axle heuristic from firing on vertical-structural cylinders. Layer (4) is critical because the word "shaft" is ambiguous — `\bshaft\b` matches both "drive shaft" (horizontal axle) and "main shaft" (vertical column). Without the veto the compiler would force-assign `axis: "x"` to the staircase's central shaft even when the LLM's Step-2 output omitted `axis` entirely. The veto keywords live in `VERTICAL_STRUCT_KEYWORDS_RE` (`column|pillar|pole|mast|newel|stem|trunk|spindle|post`), and the weak "shaft" signal now only upgrades to an axle when the brief ALSO contains wheel/hub-style parts (`briefHasWheelHubPart`). Horizontal members (axles, rungs, barrels, lenses, hubs) are unaffected because their names don't match the vertical-structural keyword set. Regressions: `npx tsx src/lib/server/modeling-briefs/test-vertical-cylinder-strip.ts` and `npx tsx src/lib/modeling-briefs/test-vertical-shaft-axis.ts`.
- **`bridgeAlong` stage pinned to `postAssembly`**: `bridgeAlong` is a constructor that CONSUMES an arranged source (the merged output of a spiral/radial/linear repeat). Ordinary constructors run during `structureGeneration`, which is scheduled before `hostGeneration` where arrange/repeat emit their merged `userData.__instances` stamp. Compiling bridgeAlong at the default stage therefore fires it before its own source exists and the runtime throws `source must resolve to an arranged/repeated group with >=2 instances (got 0)`. `compile-plan.ts` now overrides the stage for `constructorId === "bridgeAlong"` to `postAssembly`, guaranteeing it runs after every host arrange has produced its merged output regardless of how the LLM ordered its parts.
- **Vertical post alignY "top" vs "ontop" guard**: `alignY:"top"` and `alignY:"ontop"` are lexically too similar and LLMs routinely confuse them. `"top"` aligns the two tops (source top Y = target top Y), which buries a tall post below a thin host — a 0.9 m baluster with alignY:"top" on a 0.04 m tread sinks 0.86 m below the tread, and any `bridgeAlong` reading `topCenter` of the merged step+baluster tier then runs at tread height instead of baluster-top height. `"ontop"` is the one that puts source bottom at target top and makes the post stand on the host. `convertStep2Parts` in `src/lib/server/modeling-briefs/llm-only-pipeline.ts` now detects vertical structural cylinders (`baluster|spindle|post|pillar|pole|mast|newel|stem|column|shaft` in id/label) that are `attachedTo` a host via a LATERAL face (`left|right|front|back`) with `alignY:"top"` and rewrites the alignY to `"ontop"`. Axial face:"top" stacks and non-post cylinders (caps, trim) are left alone. Rule 4 in the Step-2 prompt also documents the pitfall proactively. Regression: `npx tsx src/lib/server/modeling-briefs/test-baluster-align-y.ts`.
- **Spiral/radial arrange: unconditionally strip `attachedTo` / `stackOnTop` / `stackUnder` / `snapToFace` / `alignTo` / `centerTo` / `position` from arrange params**: In `buildRepeatedPartAction` in `compile-plan.ts`, the arrange's own `anchor` + `radius` fully define WHERE each instance orbits. Any placement overlay on the arrange layer is a geometric duplicate that fights the spiral rotation. An earlier narrower invariant only stripped `attachedTo` when the seed's attach target matched the orbit anchor exactly — but LLMs routinely author a SECOND spiral (e.g. a baluster spiral) whose `targetPartId` is the central column while the seed part is attached to the tread (an intermediate, not the column). In that case the narrow check missed and the arrange re-applied a tread-width of lateral offset on every tier, so the baluster spiral diverged from the tread spiral (and a handrail threaded through the balusters followed a different curve than the treads). The compiler now deletes ALL of those placement keys from the radial/spiral arrange's params regardless of whether the attach target matches the anchor. The prototype's own seed arrange still carries the attachedTo that puts it "on orbit"; only the top-level spiral is stripped. Regression: `test-spiral-stair-compile.ts` (cases 1 and 4).
- **`stackOnTop` / `stackUnder` allowed at `hostGeneration`**: `inferAllowedStages` in `src/lib/spellshape/runtime/build-registry-operator-cards.ts` used to classify `stackUnder` as a `layoutExtractor` (and `stackOnTop` as a `placement` op), neither of which permitted `hostGeneration`. `selectStage(card, 'hostGeneration')` therefore fell through to `postHost` (or `preStructure`), which meant that when a repeat-group MEMBER has `placement.attachedTo.face = "bottom"/"top"` (lowered by `applyPlacementRelations` to a `stackUnder`/`stackOnTop` action), the placement fired AFTER the hostGeneration `mergeGroup` that consumes its output. The merge saw the member's `__attached` alias as missing and silently dropped it — on the spiral stair this left the step-support floating at the world origin outside every tread (visibly "right of" the spiral). Adding `hostGeneration` to both stack ops' allowedStages lets the existing dependency-aware ordering schedule the stack immediately before the merge within the same canonical stage. Tested alongside Fix 1 in `test-spiral-stair-compile.ts`.
- **`bridgeAlong` anchors: per-instance summit uses vertex centroid at the top/bottom band, not merged bbox center**: `_mergeInstances` in `src/lib/core/action-registry.js` stamps `userData.__instances` with `{topCenter, bottomCenter, summitTop, summitBottom, ...}`. For UNIFORM instances `topCenter` is correct, but for merged multi-member prototypes (a step tier with tread + brace + baluster) the merged bbox `topCenter` sits at the tread MIDPOINT (the tread is wide, the baluster is narrow and at the outer edge), so a handrail reading `topCenter` threads through the middle of each step instead of along the balusters. The first-pass fix traversed the instance's child meshes to find the topmost sub-mesh — which works while the instance is still a `THREE.Group` of separate Meshes. But a `mergeGroup` action flattens tread + brace + baluster into a SINGLE `BufferGeometry` before the spiral arrange runs, erasing the child hierarchy — and the child-traverse strategy then fell through to `bbox.center`, re-introducing the "handrail over tread midpoint" bug. The current implementation prefers the child-traversal summit when a genuine hierarchy exists AND falls back to a VERTEX-CENTROID summit computed directly from the BufferGeometry for single-mesh instances: it scans the `position` attribute, finds `maxY` / `minY` in world space, and returns the centroid of the vertices within ≤0.5 % of the Y-span from each extremum. For the flattened tread+baluster case the top band contains only baluster-cap vertices, so the centroid lands on the baluster top; for a symmetric uniform box the top band is the top face, so the centroid is the bbox center — same as before. `pickAnchor` in `src/lib/spellshape/geometry/bridge-along-helpers.ts` prefers `summitTop` / `summitBottom` when present. Result: handrails on a spiral-stair prototype automatically thread through the baluster tops regardless of whether the LLM authored the baluster as a separate part or as a member of the tread's `repeatGroupMemberIds` (which collapses to a single-mesh tier). Regressions: `npx tsx src/lib/core/test-summit-from-geometry.ts` (rotation and composite-mesh cases) and the `summitTop`-preference cases in `src/lib/spellshape/geometry/test-bridge-along-helpers.ts`.
- **`repeatRadial` / `repeatCircular` runtime: honor `radius: 0` and explicit `angularSpan`**: The execute bodies in `src/lib/core/action-registry.js` used `const radius = Number(params.radius) || 1;`, which coerces an INTENTIONAL `radius: 0` (the compiler's canonical "prototype is already on orbit, no extra offset" marker) to a phantom 1 m orbit because JavaScript treats 0 as falsy for `||`. Combined with a prototype whose baked geometry is already offset by ~1 m from the anchor (a tread attached to a column's right face), this doubled every tier's distance from the column and produced the "4 m wide spiral that should be 2 m wide" bug. Both copies of the generator (`repeatRadial` and its `repeatCircular` alias) now use `Number.isFinite(radiusRaw) ? radiusRaw : 1`. The same execute body also previously hard-coded `angle = (i / count) * 2π`, which silently discarded the `angularSpan` parameter forwarded by the arrange dispatcher — spirals covering less than a full revolution happened to look correct only when `(count-1)/count * 360° ≈ angularSpan`. The generator now reads `params.angularSpan` (degrees) and distributes instances across `[0, span]` inclusive when set, falling back to the historic full-circle `(i/count) * 2π` ramp when no span is given. Together these two fixes mean a brief with `radius: 0, angularSpan: 340, count: 18` produces exactly 18 instances on a 340° arc at the anchor's orbit radius, which is what `compile-plan.ts` has always intended. Covered end-to-end by `test-spiral-stair-compile.ts`.
- **Per-part action chains**: The demo2/demo3 scene graph now supports nested action chains within objects via `SceneObjectPart`. Each part contains its own `actions` array (constructor → deformer(s)), while the object-level `actions` array holds assembly actions (mergeGroup, object process, layout). The compiler (`compile-plan.ts`) builds this structure directly, eliminating the previous signature-matching hack. The UI renders parts as labeled sub-sections and provides part-level CRUD operations. Flat action arrays are still derived for execution compatibility, with visibility indices correctly accounting for parts.
- **Scene-terminal visibility**: The demo2/demo3 compiled-action UI, the 3D scene-root builder (`buildRenderRoot` in `src/lib/spellshape/action-registry-browser-executor.ts`), and any future scene-graph tooling all share a single "consumed aliases" analyzer in `src/lib/spellshape/runtime/consumed-aliases.ts`. The rule: an alias is "consumed" by a later action only when that later action TRULY transforms / merges / arranges it (`arrange.source|refs|parts`, `mergeGroup|group|mergeMeshes.parts`, or a non-constructor deformer's `inputs[0] / params.source / params.geometry`). Constructors like `bridgeAlong`, `sweep`, `extrude`, `loft`, `revolve`, `lathe`, and the `generate*` family only READ anchor data off their input (world-space bbox, `userData.__instances`, summitTop) and must never be treated as consuming it — previously the demo visibility reducer had its own sloppy version that treated every `action.inputs[]` entry as consumed, which made the 18 balusters of a spiral stair vanish from the UI the moment a handrail was added. On top of that shared rule, `initializeActionVisibility` in demo2/demo3 now applies a positive STRUCTURAL override via `collectSceneTerminalIndices` in `src/lib/spellshape/runtime/scene-terminals.ts`: the LAST action of every `SceneObjectPart.actions` chain AND of every `SceneObject.actions` top-level chain is a "scene terminal" and force-visible unless actually consumed. This matches the user's mental model — scene = array of objects, each object = groups/parts, each part = action chain, the part's LAST action is what the user added and must render by default — instead of the previous "defer to runtime `.visible`" fallback, which could hide a valid terminal just because a later unrelated action referenced its alias as a positional anchor. The reducer's priority order is: preserved > explicit `action.visible` flag > `isConsumed` (hide) > `isSceneTerminal` (show) > runtime state. Regressions: `npx tsx src/lib/spellshape/runtime/test-scene-visibility.ts` (covers bridgeAlong-does-NOT-consume, group/mergeGroup consume-their-parts, stackOnTop-consumes-inputs[0]-but-not-target, and the structural terminal-index contract across mixed part / object-level chains).
- **Cross-stage placement-dep normalization**: `compile-plan.ts` now runs `normalizeActionStagesToDeps` between `rewireBridgeAlongSources` and `orderCompiledActions`. The pass walks the flat action list to a fixpoint and bumps any action's `stage` up to ≥ the latest producer-stage of every alias in its `inputs[]` (which the emission code populates with mesh inputs AND placement targets — `stackOnTop`, `stackUnder`, inline `attachedTo.target` via `resolveConstructorParamRefs`, and the `[currentRef, targetRef]` pair on every lowered placement-relation action). Without this pass the "rover" family of bugs appeared: a chassis built with `extrude` (finisher, default `hostGeneration`) was referenced by a downstream `generateBox` (`structureGeneration`, defaults to before `hostGeneration`) carrying inline `stackOnTop=@rover__body_mainBody`. On a COLD run the chassis did not yet exist when the child fired, so the stack silently fell back to the origin; the solar panel clipped into the chassis interior and the camera lens (transparent) ended up stranded inside the opaque painted-metal body — only re-executions inheriting the prior pass's registry appeared to "work", which is why twiddling an unrelated wheel param in the UI seemed to fix the layout. The fixpoint iteration guarantees transitive propagation (e.g. `solarPanel_panel → solarPanel_mast → body_topDeck → body_mainBody` all reach `hostGeneration` in three passes), and self-references (LLM-emitted `attachedTo:<self>` on root parts) are skipped so the loop cannot infinitely bump. Face-based `attachedTo` is separately lowered by `applyPlacementRelations` to its own `hostGeneration` arrange action (which already carries the target in `inputs`), so the normalizer covers both the inline and lowered representations. Regression: `npx tsx src/lib/modeling-briefs/test-rover-stage-normalization.ts` (6 scenarios: global stage-≥-producer invariant, single-step and transitive lifts, camera chain with intermediate `__attached`, process-output `postHost` bumps, and the self-ref no-op).
- **Repeat arranges NEVER carry seed-placement params — applies to linear / grid / scatter / corner / radial / spiral / circular / surface_scatter**: `buildRepeatedPartAction` in `compile-plan.ts` now runs every variant's `placementParams` through `stripSeedPlacementLeaks` before spreading them into the arrange's own params. The seven leaked keys are `attachedTo`, `stackOnTop`, `stackUnder`, `snapToFace`, `alignTo`, `centerTo`, `position`. An earlier invariant stripped these ONLY for radial/spiral (see the note two bullets up), which missed the rover wheels bug: `wheels_axle` was attached to `body_mainBody` at `face:"right", alignY:"underside"` and carried a `linear_repeat` of count=2 along z. The seed was correctly placed by its own `applyPlacementRelations`-emitted `__attached` arrange at `hostGeneration`, but the final `arrange linear` — because the `linear_repeat` branch spread `...placementParams` verbatim — re-applied the same `rightAttach` positioning onto the already-positioned 2-copy cluster, migrating the whole wheel pair to the chassis's right face instead of fanning it along z under the body. Generalising the strip to every repeat type eliminates the class of bugs entirely: the seed's own lowered placement action is the SINGLE source of truth for "where the prototype lives in world space", and the repeat's own `anchor` / `target` / `axis` / `spacing` / `count` / `radius` / `angularSpan` / `areaWidth` / `areaDepth` are the single source of truth for "how copies are distributed around it". `rotation` is intentionally preserved — per-instance rotation lives on `rotationPerIndexDeg` and seed rotation is baked into the prototype mesh.
